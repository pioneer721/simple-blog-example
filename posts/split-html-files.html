Why I split my blog into multiple HTML files

April 23, 2026 · 4 min read

When I first started, I dumped everything into a single index.html. The home page, every post, the about section, all in one file. It worked, but the file was getting over a thousand lines, and editing one post meant scrolling past everything else.

So I split it up. Here is the structure I settled on.

simple-blog-example/
├── index.html
├── posts/
│   ├── hello-world.html
│   └── split-html-files.html
└── css/
    └── style.css

Why this is better

Edits are local. Changing one post only touches one file. Less risk of breaking something else.

Each post has a real URL. /posts/hello-world.html is shareable. With a single file, you have to use anchor links, which are fragile.

Browsers load faster. Each page only carries its own content. The shared CSS file gets cached after the first visit, so navigating between posts feels instant.

It scales. Adding a new post means creating one new file in posts/ and adding one card to the home page. That is it.

The one thing to watch out for

If you want the same header and footer on every page, you have to copy the markup into each file. That is a bit of duplication. Two ways to handle it:

  • Copy and paste. Fine for a small blog. Update all files if the header changes.
  • Load with JavaScript. Put the header in header.html and fetch it into each page. Cleaner, but adds a tiny bit of complexity.

For under twenty posts, copy and paste is simpler and works great.
